Section 2

The DOCTYPE Declaration

The DOCTYPE declaration is the very first line of every HTML document. It is not a tag — it is an instruction to the browser telling it which version of HTML the document uses.

DOCTYPE syntax
1<!DOCTYPE html>
Standards Mode

When DOCTYPE is present, browsers render the page in standards mode — following the HTML specification exactly as intended.

Quirks Mode (Avoid)

Without DOCTYPE, browsers fall back to Quirks Mode — a legacy compatibility mode with inconsistent, unpredictable rendering behaviour.

Always First

DOCTYPE must appear as the very first line of the document. Nothing — not even a blank line — should appear before it.

HTML5 Simplified

Older HTML versions (4.01, XHTML) required long DOCTYPE strings. HTML5 simplified it to just <!DOCTYPE html>.

Never Skip the DOCTYPEOmitting the DOCTYPE causes browsers to switch to Quirks Mode. Your layout, spacing, and rendering may break in unexpected ways that are very difficult to debug. Always include it as your very first line.
Section 3

The <html> Element

The <html> element is the root element of every HTML document. Think of it as the outer container that holds absolutely everything else — the <head>, the <body>, and every element inside them.

Without lang attribute
1<html>
2  <!-- content -->
3</html>
With lang attribute (Best Practice)
1<html lang="en">
2  <!-- content -->
3</html>
Why the lang Attribute Matters
Accessibility
Screen readers use it to select the correct pronunciation engine
SEO
Search engines serve your page to the correct regional audience
Translation
Browser translation tools detect source language automatically
Section 4

The <head> Section

The <head> section is the control room of your webpage. It contains metadata — information about the page that the browser uses but does not display to the user directly.

What Lives Inside <head>
<title> — Browser tab title
<meta> — Charset, viewport, description
<link> — Stylesheets, favicons
<style> — Internal CSS styles
<script> — JavaScript references
Headings, paragraphs, images — these belong in <body>
A Very Common MistakeNever place visible content like headings or paragraphs inside the <head>. That content belongs in <body>. The head is for metadata only.
Section 5

The <body> Section

The <body> section is where everything a user actually sees is written. If it appears on the screen — headings, text, images, buttons, tables, forms — it belongs inside body.

Body section example
1<body>
2
3  <h1>Hello World</h1>
4
5  <p>This is my first webpage.</p>
6
7</body>
Head vs Body — Think of a Document

The <head> is like the properties panel of a document — title, author, settings — information about the document that you configure but do not see in the printed output. The <body> is the actual printed page — every word, image, table, and heading a reader sees. One is metadata. One is content. Never confuse which belongs where.

Section 6

The <title> Element

The <title> element defines the text shown on the browser tab. It is also the first thing search engines display in search results. A meaningful title matters more than beginners often realise.

title element
1<title>HTML Mastery</title>
Poor Title
<title>Home</title>
Good Title
<title>HTML Mastery | Chapter 2</title>
Why a Good Title Matters
User Experience — Users with many tabs open need to identify your page instantly.
SEO — Google displays the title as the clickable link in search results.
Bookmarks — When saved to favourites, the title becomes the bookmark name.
Section 7

Meta Tags

Meta tags are self-closing elements placed inside <head> that provide additional information about the webpage. They affect encoding, responsiveness, and search engine behaviour.

Character Encoding
meta charset
1<meta charset="UTF-8">

UTF-8 is the universal character encoding standard. It ensures that special characters, foreign language letters, and symbols display correctly in all browsers across all operating systems.

Responsive Design (Viewport)
meta viewport
1<meta name="viewport" content="width=device-width, initial-scale=1.0">

This meta tag tells mobile browsers to set the viewport width equal to the device screen width and display the page at a 1:1 scale. Without it, mobile browsers zoom out and show a tiny, unreadable desktop layout.

Meta Description (SEO)
meta description
1<meta name="description" content="Learn HTML from scratch.">

The meta description appears as the summary text beneath your page title in Google search results. A well-written description improves click-through rates significantly.

Always Include These ThreeFor every HTML page you build, include charset, viewport, and a descriptive title as a minimum. These three lines alone make your page encoded correctly, mobile-friendly, and search-engine visible.
Section 8

HTML Comments

HTML comments are notes written inside your code that the browser completely ignores. They are visible only to developers reading the source code. Every professional developer uses them.

Comment syntax
1<!-- This is an HTML comment -->
2
3<!-- ====== NAVIGATION SECTION ====== -->
4<nav></nav>
5
6<!-- This heading will be replaced later -->
7<h1>Placeholder Title</h1>
Explain Code

Leave notes for yourself and teammates explaining what a section does.

Organise Sections

Use comments as visual dividers to separate header, nav, main, and footer areas.

Team Collaboration

Help other developers understand your intent when working on shared projects.

Invisible Output

Comments never appear on the rendered webpage. They are for developers only.

Section 9

Whitespace in HTML

HTML collapses all consecutive whitespace — spaces, tabs, and line breaks — into a single space when rendering output. This behaviour surprises many beginners. Understanding it prevents frustrating layout bugs.

Code Written
1<p>Hello        World</p>
Browser Output
Hello World
Extended Whitespace Example
whitespace-demo.html
1<p>
2    This text
3
4    spans multiple
5
6    lines in the code.
7</p>
Browser renders this as:
This text spans multiple lines in the code.
Whitespace is for Developers, Not the BrowserYou can use as many spaces, tabs, and blank lines as you like in your HTML source code. This helps you write clean, readable code. The browser collapses it all anyway, so indentation costs you nothing in terms of output.
Section 10

HTML Case Insensitivity

HTML tags are not case-sensitive. <h1>, <H1>, and <H1> are all treated identically by the browser. However, one convention is universally adopted for good reason.

Avoid Uppercase
<HTML>
<HEAD>
  <TITLE>My Page</TITLE>
</HEAD>
<BODY>
  <H1>Hello</H1>
</BODY>
</HTML>
Always Lowercase
<html>
<head>
  <title>My Page</title>
</head>
<body>
  <h1>Hello</h1>
</body>
</html>
Lowercase Is the Universal StandardThe W3C (the organisation that writes the HTML specification) recommends lowercase. Modern frameworks, CSS preprocessors, validators, and team style guides all expect lowercase. Make it a habit from day one and you will never need to fix it later.

Watch Out

Common Beginner Mistakes

Forgetting DOCTYPE
Starting the file with <html> without any DOCTYPE declaration on line 1.
Always write <!DOCTYPE html> as the very first line, before anything else.
Content Inside Head
Placing headings, paragraphs, or images inside the <head> section expecting them to appear on screen.
All visible content belongs exclusively inside <body>. The head is for metadata only.
Metadata in Body
Placing <meta> tags, the <title>, or <link> elements inside <body>.
All meta tags, title, and stylesheet links belong in the <head> section, never in body.
Inconsistent Tag Case
Mixing uppercase and lowercase tags inconsistently: <HEAD> here, <body> there.
Write all HTML tags in lowercase consistently. This is the industry convention and makes code readable.
Missing lang Attribute
Writing <html> without a lang attribute on the root element.
Always add lang="en" (or your document language) to the <html> tag for accessibility and SEO.
Vague Page Titles
Using generic titles like <title>Home</title> or <title>Page</title> that give no meaningful information.
Write descriptive, specific titles that tell users and search engines exactly what the page is about.
Interview Prep

Interview Questions

These questions are asked in college exams and real developer interviews. Think through your answer before expanding each one.

The DOCTYPE declaration tells the browser which version of HTML the document uses. For HTML5 it is simply <!DOCTYPE html>. Without it the browser enters Quirks Mode, a legacy compatibility mode with inconsistent rendering behaviour. It must always appear as the very first line of the document.
The <html> element is the root element. It wraps the entire HTML document — every other element, including <head> and <body>, exists inside it. It should always include a lang attribute specifying the document's language.
The <head> section contains metadata — information about the page that is not displayed to the user, such as the title, character encoding, viewport settings, and stylesheet links. The <body> section contains all visible content: headings, paragraphs, images, links, tables, and forms that users see and interact with.
The <title> element defines the text shown on the browser tab. It is also what Google displays as the clickable headline in search results, and what browsers use as the default bookmark name. A meaningful, descriptive title improves user experience, SEO ranking, and navigation when multiple tabs are open.
Meta tags are self-closing HTML elements placed inside the <head> section. They provide additional information about the webpage to browsers and search engines. Common meta tags include charset (character encoding), viewport (responsive design), and description (SEO summary). They are not displayed on the page itself.
The viewport meta tag <meta name="viewport" content="width=device-width, initial-scale=1.0"> instructs mobile browsers to set the viewport width to match the device screen width and display at a natural 1:1 scale. Without it, mobile browsers zoom out and render a shrunken desktop layout that is unreadable on small screens.
HTML comments are developer notes written inside the source code that are completely ignored by the browser. They do not appear in the rendered output. The syntax is <!-- comment text here -->. They are used to explain code, organise sections with labels, and communicate intent to other developers.
HTML collapses all consecutive whitespace — multiple spaces, tabs, and line breaks — into a single space in the rendered output. This means you can freely use indentation and blank lines to make your source code readable without affecting what the browser displays. The browser only renders one space regardless of how many you write.
HTML is not case-sensitive. The browser treats <H1> and <h1> identically. However, the W3C specification and all modern development conventions recommend writing all HTML tags in lowercase. This ensures compatibility with XHTML (which is case-sensitive), maintains consistency, and produces cleaner, more readable code.
Lowercase is the recommended convention by the W3C. It ensures compatibility with XHTML, which is case-sensitive. It also makes code more readable and consistent, aligns with professional team style guides, and is what all modern frameworks and validators expect. Building the lowercase habit from the beginning prevents problems when moving to more advanced technologies.
